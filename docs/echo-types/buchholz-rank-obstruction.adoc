= Buchholz `_<бөҮ_` rank-embedding obstruction
:date: 2026-05-20
:toc:

This note records why the natural rank-embedding route to unbudgeted
`WellFounded _<бөҮКіб¶ _` (and to `WellFounded _<бөҮвҒә_`) is *structurally
impossible* for the `Ordinal.Buchholz.Order._<бөҮ_` relation as
currently defined. The relation is well-founded вҖ” `Ordinal.Buchholz.
WellFounded.wf-<бөҮ` proves it вҖ” but it is *not ordinally sound*, so no
rank function `BT вҶ’ Ord` can be strict-monotone on every constructor.

This note supersedes the "Route B (recommended next attempt)" framing
in `docs/echo-types/buchholz-extended-wf.md` and the "Phase 2.2 open
lemma" framing in `Ordinal.Buchholz.RankBrouwer.agda`. Both are
historical now; the rank-embedding route is closed.

== The obstruction in one counterexample

The constructor `<бөҮ-О©+` of `_<бөҮ_`:

[source,agda]
----
<бөҮ-+О©  : вҲҖ {x y Ој} вҶ’ x <бөҮ bOmega Ој вҶ’ bplus x y <бөҮ bOmega Ој
----

(`Ordinal.Buchholz.Order.agda`, line 82.)

This says: if `x` is `<бөҮ`-below the О©-marker, then so is `bplus x y`
*for any `y`*. Concretely, instantiate `Ој = fin 0`, `x = bzero`,
`y = bOmega (fin 1)`:

[source,agda]
----
witness : bplus bzero (bOmega (fin 1)) <бөҮ bOmega (fin 0)
witness = <бөҮ-+О© <бөҮ-0-О©
----

The premise `<бөҮ-0-О© : bzero <бөҮ bOmega (fin 0)` is a constructor of
`_<бөҮ_`. So the witness is derivable.

Any additive rank `rank : BT вҶ’ Ord` of the SA-recommended shape
(`Ordinal.Buchholz.RankBrouwer.agda`):

[source,agda]
----
rank bzero        = oz
rank (bOmega ОҪ)   = Пү-rank ОҪ
rank (bplus x y)  = rank x вҠ• rank y
rank (bpsi ОҪ Оұ)   = psi-rank ОҪ (rank Оұ)
----

computes the LHS of the witness to `oz вҠ• Пү-rank (fin 1) = two` and
the RHS to `Пү-rank (fin 0) = one` (in Brouwer notation). The required
`rank-mono` would force `two <вҖІ one`, which is `osuc (osuc oz) вүӨвҖІ
osuc oz` вҖ” reducing to `osuc oz вүӨвҖІ oz`, which is `вҠҘ`. So `rank-mono`
is refuted at a checked counterexample.

== Why no rank shape rescues it

Sufficient conditions per `_<бөҮ_` constructor on a candidate
`rank-+ : BT вҶ’ BT вҶ’ Ord` for the `bplus` head:

* `<бөҮ-+О© x<О©`: need `rank-+ x y <вҖІ rank-О© Ој` given `rank x <вҖІ rank-О©
  Ој`. *The right summand `y` is arbitrary*, so `rank-+ x y` must be
  bounded by something that doesn't grow with `y`. Equivalently:
  `rank-+ x y` cannot depend on `y` in any unbounded way.
* `<бөҮКіб¶ -+2` (and `<бөҮвҒә-+2`): need `rank-+ x y <вҖІ rank-+ x z` given
  `rank y <вҖІ rank z`. So `rank-+ x y` *must* be strict-monotone in
  `y`.

These two requirements are jointly incompatible: monotone-in-`y`
forces `rank-+ x y` to grow with `y`; the `<бөҮ-+О©` bound forces it not
to. The same tension appears for `<бөҮ-О©+`/`<бөҮ-ПҲ+` versus `<бөҮ-+1`.

No additive, multiplicative, or constructive ordinal arithmetic on
`rank x` and `rank y` resolves this. Constant-in-`y` shapes (e.g.
`rank-+ x y = rank x`) lose `<бөҮКіб¶ -+2`; bounded-in-`y` shapes (e.g.
`rank-+ x y = max (rank x) (rank y) + 1`) fail `<бөҮ-+О©` when `y` is
ordinally above `Ој`.

The structural root cause is that `<бөҮ-+О©` (and its companion
constructors) make `_<бөҮ_` a *syntactic* order, not the genuine
ordinal `<` on Cantor normal forms вҖ” they are sound only when both
summands are constrained to be `<бөҮ`-below `Ој`, which is exactly the
constraint that `Ordinal.Buchholz.WellFormed` would add on top.
`_<бөҮ_` itself does not require well-formedness, so the rank route
cannot succeed at the level of `_<бөҮ_` alone.

== What `_<бөҮ_` is and isn't

`_<бөҮ_` is a *syntactic* strict order on raw `BT` terms, chosen so
that the constructor-by-constructor pattern of `wf-<бөҮ`
(`Ordinal.Buchholz.WellFounded`) closes by direct accessibility.
`wf-<бөҮ` doesn't need ordinal soundness вҖ” it only needs every
predecessor chain to ground out structurally, which it does via
the О©-bundle and bzero-base-case structure.

`_<бөҮ_` is *not* the ordinal order on Cantor normal forms. The
constructors `<бөҮ-+О©`/`<бөҮ-О©+`/`<бөҮ-+ПҲ`/`<бөҮ-ПҲ+` admit derivations
whose ordinal semantics is the reverse (or unrelated). To recover
ordinal soundness one would have to restrict to *well-formed* BT
terms (the predicate stubbed in `Ordinal.Buchholz.WellFormed`),
where `bplus Оұ ОІ` is constrained to be a CNF tail with `ОІ вүӨ Оұ`.

Under that restriction `<бөҮ-+О©` becomes sound: `x <бөҮ О©_Ој` with `y вүӨ
x` gives `y <бөҮ О©_Ој` and therefore `x + y <бөҮ О©_Ој`. But the
restriction propagates: `<бөҮ-trans (<бөҮ-+1 _) (<бөҮ-+О© _)` no longer
type-checks at raw BT, because the conclusion's predecessor
`bplus xвӮҒ xвӮӮ` would need its own WF constraint that the chain
doesn't currently supply. The full re-ordering on WF-restricted BT
is *substantially* larger work than the rank-embedding route the
roadmap originally framed.

1. *Promote `_<бөҮ_` to a WF-restricted relation*. Re-state every

   every existing client. The "eliminate the в„• budget" goal in
   `roadmap.md` should be retired or recast.

== Slices 1вҖ“5 of the Пү-power unblock (2026-05-20 evening)

After the analysis above was written, an explicit five-slice
infrastructure was landed that closes 9 of the 13 constructors' rank-
mono cases via the WfCNF-restricted Buchholz order `_<бөҮвҒ»_`
(`Ordinal.Buchholz.OrderRestricted`).  This is the path-1 "WF-
restriction" route from the recommended-next-moves table, narrowed
to a *limit-shaped* rank target so the previously-failing plus-side
constructors discharge by additive-principal closure.

=== What landed

* *Slice 1* (`Ordinal.Brouwer.OmegaPow`) вҖ” definitions: finite
  iterated product `_В·в„•_`, Пү-powers `Пү^_`, basic identification
  lemmas, one-step strict-monotonicity `Пү^-strict-mono-suc`.

  `Ordinal.Brouwer.Phase13`), `В·в„•-mono-вүӨ-left`, and the general gap
  strict-mono `Пү^-strict-mono : m < n вҶ’ Пү^ m <вҖІ Пү^ n`.
* *Slice 3* вҖ” вҠ•-associativity (`вҠ•-assoc-вүӨ` / `вҠ•-assoc-вүҘ` in
  Phase13; both directions, funext-free `вүӨвҖІ` form), the bridge
