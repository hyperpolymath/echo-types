= Buchholz `_<·µá_` rank-embedding obstruction
:date: 2026-05-20
:toc:

This note records why the natural rank-embedding route to unbudgeted
`WellFounded _<·µá ≥·∂†_` (and to `WellFounded _<·µá‚Å∫_`) is *structurally
impossible* for the `Ordinal.Buchholz.Order._<·µá_` relation as
currently defined. The relation is well-founded ‚Äî `Ordinal.Buchholz.
WellFounded.wf-<·µá` proves it ‚Äî but it is *not ordinally sound*, so no
rank function `BT ‚Üí Ord` can be strict-monotone on every constructor.

This note supersedes the "Route B (recommended next attempt)" framing
in `docs/echo-types/buchholz-extended-wf.md` and the "Phase 2.2 open
lemma" framing in `Ordinal.Buchholz.RankBrouwer.agda`. Both are
historical now; the rank-embedding route is closed.

== The obstruction in one counterexample

The constructor `<·µá-Œ©+` of `_<·µá_`:

[source,agda]
----
<·µá-+Œ©  : ‚àÄ {x y Œº} ‚Üí x <·µá bOmega Œº ‚Üí bplus x y <·µá bOmega Œº
----

(`Ordinal.Buchholz.Order.agda`, line 82.)

This says: if `x` is `<·µá`-below the Œ©-marker, then so is `bplus x y`
*for any `y`*. Concretely, instantiate `Œº = fin 0`, `x = bzero`,
`y = bOmega (fin 1)`:

[source,agda]
----
witness : bplus bzero (bOmega (fin 1)) <·µá bOmega (fin 0)
witness = <·µá-+Œ© <·µá-0-Œ©
----

The premise `<·µá-0-Œ© : bzero <·µá bOmega (fin 0)` is a constructor of
`_<·µá_`. So the witness is derivable.

Any additive rank `rank : BT ‚Üí Ord` of the SA-recommended shape
(`Ordinal.Buchholz.RankBrouwer.agda`):

[source,agda]
----
rank bzero        = oz
rank (bOmega ŒΩ)   = œâ-rank ŒΩ
rank (bplus x y)  = rank x ‚äï rank y
rank (bpsi ŒΩ Œ±)   = psi-rank ŒΩ (rank Œ±)
----

computes the LHS of the witness to `oz ‚äï œâ-rank (fin 1) = two` and
the RHS to `œâ-rank (fin 0) = one` (in Brouwer notation). The required
`rank-mono` would force `two <‚Ä≤ one`, which is `osuc (osuc oz) ‚â§‚Ä≤
osuc oz` ‚Äî reducing to `osuc oz ‚â§‚Ä≤ oz`, which is `‚ä•`. So `rank-mono`
is refuted at a checked counterexample.

restriction or a non-additive measure.**

The four passing cases are formalised as a partial Agda artifact
in `proofs/agda/Ordinal/Buchholz/RankPartial.agda` (2026-05-20):

  time.

This is a *stepping stone*: the four lemmas are reusable for both
unblock paths.  Path 1 (WF-restriction) needs to additionally
discharge the nine failing cases under a CNF tail-constraint;
the four passing cases survive the restriction unchanged.  Path 2
(non-additive denotational measure) needs to replace the rank
shape entirely; the four passing cases serve as sanity checks
that the new shape doesn't regress.

== Why no rank shape rescues it

Sufficient conditions per `_<·µá_` constructor on a candidate
`rank-+ : BT ‚Üí BT ‚Üí Ord` for the `bplus` head:

* `<·µá-+Œ© x<Œ©`: need `rank-+ x y <‚Ä≤ rank-Œ© Œº` given `rank x <‚Ä≤ rank-Œ©
  Œº`. *The right summand `y` is arbitrary*, so `rank-+ x y` must be
  bounded by something that doesn't grow with `y`. Equivalently:
  `rank-+ x y` cannot depend on `y` in any unbounded way.
* `<·µá ≥·∂†-+2` (and `<·µá‚Å∫-+2`): need `rank-+ x y <‚Ä≤ rank-+ x z` given
  `rank y <‚Ä≤ rank z`. So `rank-+ x y` *must* be strict-monotone in
  `y`.

These two requirements are jointly incompatible: monotone-in-`y`
forces `rank-+ x y` to grow with `y`; the `<·µá-+Œ©` bound forces it not
to. The same tension appears for `<·µá-Œ©+`/`<·µá-œà+` versus `<·µá-+1`.

No additive, multiplicative, or constructive ordinal arithmetic on
`rank x` and `rank y` resolves this. Constant-in-`y` shapes (e.g.
`rank-+ x y = rank x`) lose `<·µá ≥·∂†-+2`; bounded-in-`y` shapes (e.g.
`rank-+ x y = max (rank x) (rank y) + 1`) fail `<·µá-+Œ©` when `y` is
ordinally above `Œº`.

The structural root cause is that `<·µá-+Œ©` (and its companion
constructors) make `_<·µá_` a *syntactic* order, not the genuine
ordinal `<` on Cantor normal forms ‚Äî they are sound only when both
summands are constrained to be `<·µá`-below `Œº`, which is exactly the
constraint that `Ordinal.Buchholz.WellFormed` would add on top.
`_<·µá_` itself does not require well-formedness, so the rank route
cannot succeed at the level of `_<·µá_` alone.

== What `_<·µá_` is and isn't

`_<·µá_` is a *syntactic* strict order on raw `BT` terms, chosen so
that the constructor-by-constructor pattern of `wf-<·µá`
(`Ordinal.Buchholz.WellFounded`) closes by direct accessibility.
`wf-<·µá` doesn't need ordinal soundness ‚Äî it only needs every
predecessor chain to ground out structurally, which it does via
the Œ©-bundle and bzero-base-case structure.

`_<·µá_` is *not* the ordinal order on Cantor normal forms. The
constructors `<·µá-+Œ©`/`<·µá-Œ©+`/`<·µá-+œà`/`<·µá-œà+` admit derivations
whose ordinal semantics is the reverse (or unrelated). To recover
ordinal soundness one would have to restrict to *well-formed* BT
terms (the predicate stubbed in `Ordinal.Buchholz.WellFormed`),
where `bplus Œ± Œ≤` is constrained to be a CNF tail with `Œ≤ ‚â§ Œ±`.

Under that restriction `<·µá-+Œ©` becomes sound: `x <·µá Œ©_Œº` with `y ‚â§
x` gives `y <·µá Œ©_Œº` and therefore `x + y <·µá Œ©_Œº`. But the
restriction propagates: `<·µá-trans (<·µá-+1 _) (<·µá-+Œ© _)` no longer
type-checks at raw BT, because the conclusion's predecessor
`bplus x‚ÇÅ x‚ÇÇ` would need its own WF constraint that the chain
doesn't currently supply. The full re-ordering on WF-restricted BT
is *substantially* larger work than the rank-embedding route the
roadmap originally framed.

== Concrete landing this session

The shared-binder shapes `<·µá‚Å∫-œàŒ±`/`<·µá‚Å∫-+2` now have a *budgeted*
WF carrier `_<·µá‚Å∫·µá_` in `Ordinal.Buchholz.OrderExtendedBudget`,
mirroring `_<·µá ≥·∂†·µá_` in `Ordinal.Buchholz.RecursiveSurfaceBudget`.
Downstream consumers can switch to the budgeted relation without
losing well-foundedness. Pinned in `Smoke.agda`; wired into
`All.agda`.

== Recommended next moves (each is substantial)

1. *Promote `_<·µá_` to a WF-restricted relation*. Re-state every
   constructor with explicit `WellFormed` premises, re-prove
   transitivity and `wf-<·µá` under the restriction, and then rank
   into Brouwer or directly. Substantial: the 13-constructor
   matrix + the inversion lemmas all need restating. Likely 2‚Äì3
   weeks of disciplined proof work.

2. *Non-additive denotational measure*. Replace the Brouwer-rank
   shape with a function `BT ‚Üí Œ±` for some target `Œ±` whose order
   respects all of `_<·µá_`'s admitted constructors including the
   syntactic `<·µá-+Œ©` bridge. This is essentially "give a
   denotational semantics to `_<·µá_` that respects its syntactic
   quirks", which is at least as hard as the original Buchholz WF
   problem.

3. *Accept the budgeted form as canonical* and document. The
   downstream consumers (`VeblenComparisonModel`,
   `IteratedExtendedOrder`) already work with the depth-stratified
   tower; the budgeted `_<·µá ≥·∂†·µá_` and `_<·µá‚Å∫·µá_` are sufficient for
   every existing client. The "eliminate the ‚Ñï budget" goal in
   `roadmap.md` should be retired or recast.

== Slices 1‚Äì5 of the œâ-power unblock (2026-05-20 evening)

After the analysis above was written, an explicit five-slice
infrastructure was landed that closes 9 of the 13 constructors' rank-
mono cases via the WfCNF-restricted Buchholz order `_<·µá‚Åª_`
(`Ordinal.Buchholz.OrderRestricted`).  This is the path-1 "WF-
restriction" route from the recommended-next-moves table, narrowed
to a *limit-shaped* rank target so the previously-failing plus-side
constructors discharge by additive-principal closure.

=== What landed

* *Slice 1* (`Ordinal.Brouwer.OmegaPow`) ‚Äî definitions: finite
  iterated product `_¬∑‚Ñï_`, œâ-powers `œâ^_`, basic identification
  lemmas, one-step strict-monotonicity `œâ^-strict-mono-suc`.
* *Slice 2* ‚Äî left-monotonicity of `_‚äï_` (`‚äï-mono-‚â§-left` in
  `Ordinal.Brouwer.Phase13`), `¬∑‚Ñï-mono-‚â§-left`, and the general gap
  strict-mono `œâ^-strict-mono : m < n ‚Üí œâ^ m <‚Ä≤ œâ^ n`.
* *Slice 3* ‚Äî ‚äï-associativity (`‚äï-assoc-‚â§` / `‚äï-assoc-‚â•` in
  Phase13; both directions, funext-free `‚â§‚Ä≤` form), the bridge
  `¬∑‚Ñï-add-‚â§ : Œ±¬∑‚Ñïk ‚äï Œ±¬∑‚Ñïm ‚â§‚Ä≤ Œ±¬∑‚Ñï(m+k)`, and the keystone
  *additive-principal* lemma `Œ± <‚Ä≤ œâ^(suc n) ‚Üí Œ≤ <‚Ä≤ œâ^(suc n) ‚Üí
  Œ± ‚äï Œ≤ <‚Ä≤ œâ^(suc n)`.
* *Slice 4* (`Ordinal.Buchholz.RankPow`) ‚Äî `œâ-rank-pow : OmegaIndex
  ‚Üí Ord` (limit-shaped: `fin n ‚Ü¶ œâ^(suc n)`), `rank-pow : BT ‚Üí Ord`
  using `œâ-rank-pow`, positivity `œâ-rank-pow-pos`, monotonicity
  `œâ-rank-pow-mono`, and reusable compositional primitives
  `rank-pow-bplus-right-mono` / `rank-pow-via-left` /
  `rank-pow-bplus-into-œâ-rank-pow`.
* *Slice 5* ‚Äî 9 of 13 per-constructor rank-mono lemmas exposed
  *relation-agnostically* in `Ordinal.Buchholz.RankPow`:
  `rank-mono-<·µá-0-Œ©`, `rank-mono-<·µá-0-œà`, `rank-mono-<·µá-Œ©Œ©`,
  `rank-mono-<·µá-Œ©œà`, `rank-mono-<·µá-œàŒ©`, `rank-mono-<·µá-Œ©+`,
  `rank-mono-<·µá-œà+`, `rank-mono-<·µá-+Œ©`, `rank-mono-<·µá-+œà`.  Each
  states a pure rank inequality conclusion from pure rank-inequality
  hypotheses; consumers (whether `_<·µá‚Åª_` or `_<·µá ≥·∂†_`) recurse on
  their relation's constructor and apply the matching primitive.

=== What remains open

* *œà-admissibility predicate* (`WellFormedAdmissible.agda`, ~80 lines).
  Refines `WfCNF` with the constraint that œà-arguments live in their
  binder's Œ©-closure `C_ŒΩ`.  Closes `<·µá-œàŒ±` and `<·µá-œàŒ©‚â§` once the
  rank-pow on `bpsi ŒΩ Œ±` is refined to `œâ-rank-pow ŒΩ ‚äï rank-pow Œ±`
  under the admissibility carrier.
* *`<·µá-+1` joint-bplus*.  Needs either a coarser dominator (e.g., a
  function `BT ‚Üí OmegaIndex` returning the leading Œ©-index, then
  rank into `œâ-rank-pow ‚àò leading-index`) or a richer rank shape.
  The remaining challenge in this constructor is the structural
  asymmetry: `rank-pow (bplus z‚ÇÅ z‚ÇÇ)` lives "outside" the additive-
  principal closure, so the standard œâ-power closure argument
  doesn't apply directly.
* *Mutual recursion for `rank-pow-mono-<·µá‚Åª`*.  The case-specific
  primitives above are individually proven, but the umbrella theorem
  `x <·µá‚Åª y ‚Üí rank-pow x <‚Ä≤ rank-pow y` needs a mutual `<·µá` / `‚â§·µá`
  recursion that pattern-matches on the source's WfCNF carrier to
  derive the tail bound `y ‚â§·µá x ‚Üí rank-pow y ‚â§‚Ä≤ rank-pow x`.  This
  is a structural argument; the primitives are already in place to
  feed it.  Deferred pending the `<·µá-+1` resolution since that
  constructor is required for the case-completeness of the umbrella.

=== Status note

The "rank-embedding route is closed" framing from this document's
introduction is *narrowed*: it remains true for the unrestricted
`_<·µá_`, but the WfCNF restriction `_<·µá‚Åª_` together with the limit-
shaped rank-pow discharges 9 of the 13 constructor cases.  The
remaining 3 constructors have well-understood structural blockers
(œà-admissibility for two; joint-bplus structural-rank for one)
rather than the fundamental obstruction the original counterexample
exhibited.
